PriorAuthAI Governance Console — Help Guide

This sandbox demonstrates a fully governed AI agent for healthcare prior authorization. It shows how Skills, Rules, Hooks, and Progressive Disclosure work together to create a transparent, auditable, and safe clinical decision support system — all powered by declarative markdown that non-programmers can edit.

🏥 The Use Case: AI-Powered Prior Authorization

Prior authorization (PA) is the process where health insurers review whether a requested medical procedure or medication meets clinical necessity guidelines before approving coverage. Traditionally, this involves manual review by clinical staff — slow, inconsistent, and expensive.

PriorAuthAI automates this review using an AI agent that follows strict governance guardrails. It evaluates clinical notes against payer-defined policy criteria (expressed in OPA Rego rules) and either auto-approves cases that clearly meet guidelines or escalates ambiguous cases to a human Medical Director — never auto-denying.

Key Principles

  • Clinical Conservatism — The agent never denies. Uncertain cases escalate to humans.
  • Full Transparency — Every decision step is traced and auditable in real-time.
  • PHI Protection — Patient data is redacted from logs before any external processing.
  • Declarative Policies — Clinical directors edit markdown, not code.
  • Dynamic Extensibility — New skills can be added at runtime via plain English.

🛠️ Skills — The Agent's Capabilities

Skills are discrete, composable functions that the agent invokes during its review pipeline. Each skill has a defined contract: specific inputs it requires and outputs it produces. Skills are declared in skills_declaration.md (human-readable) and compiled into executable JavaScript in skills.js.

Core Skills

SKILL PURPOSE PIPELINE STAGE
VerifyCoverageSkill Checks member enrollment and plan coverage for the requested service category Stage 2: Coverage
RetrieveGuidelinesSkill Fetches the clinical policy rules for the requested CPT procedure code Stage 2: Coverage
ExtractClinicalDataSkill Parses provider clinical notes to extract structured evidence (symptom duration, therapy weeks, findings) Stage 3: OPA Rego
EvaluateClinicalCriteriaSkill Maps extracted evidence against OPA Rego policy checklist criteria Stage 3: OPA Rego
GenerateDecisionNoticeSkill Drafts the formal patient/provider correspondence letter with citations Stage 5: Notice
VerifyNpiStatusSkill Dynamically injected skill that validates provider NPI registry format Dynamic (post-coverage)

Creating Skills from Plain English

The "Create Skill from Plain English" panel allows non-programmers to describe a new validation rule in natural language. The system parses the description to extract:

  • Action verb → becomes the skill name prefix (Check, Verify, Flag, etc.)
  • Domain objects → determine inputs (age→patientDob, NPI→providerNpi)
  • Conditions → numeric comparisons ("over 65"), text matching ("contains opioid")
  • Actions → what happens when conditions are met ("flag for senior review")

The generated skill is security-audited by SkillJuror, appended to both the markdown declaration and the executable JavaScript, then immediately available in the agent pipeline.

⚙️ Rules — Governance Guardrails

Rules are non-negotiable governance constraints enforced at specific lifecycle points. Unlike skills (which do work), rules act as guardrails — they validate, constrain, and transform data to ensure compliance. Each rule can be independently toggled on/off via the UI checkboxes.

RULE ENFORCEMENT WHEN
RULE-01: PHI Redaction Scrubs SSN, DOB, and patient names from all trace logs before display or storage on_request_received
RULE-02: Clinical Conservatism Forces escalation to human Medical Director for any unmet or ambiguous criteria — never auto-denies on_criteria_evaluated
RULE-03: Citation Compulsory Validates that every notice letter cites the specific policy ID and criteria references on_notice_generated
RULE-04: Code Match Validates CPT/ICD-10 code formatting and confirms the diagnosis maps to the procedure's guidelines on_guidelines_loaded
RULE-05: Plain Language Translates medical acronyms (MRI, CPT, PT, DMARD) to patient-readable language in notice letters on_notice_generated

OPA Rego Rules Engine

Clinical policy criteria are expressed in OPA Rego (Open Policy Agent) format — a declarative policy language. Medical directors edit the human-readable rules_declaration.md markdown file, and the SkillJuror compiler automatically generates the corresponding rules.rego file. This means:

  • Non-programmers maintain policy in plain markdown
  • The Rego interpreter evaluates criteria deterministically
  • Adding new policies requires zero code changes
  • All criteria thresholds (symptom weeks, therapy duration) are data-driven

📄 UHC Policy PDF — The Source of Truth

The file real_payer_policy_uhc.pdf is a real-world payer clinical policy document that serves as the authoritative source from which all rules and guidelines in this system are derived. It represents how actual health insurance payers (like UnitedHealthcare) publish their medical necessity criteria.

How the PDF Guides the System

The workflow from payer policy to automated enforcement follows a progressive chain:

Agent Evaluation (Automated Decision)

The Rego interpreter evaluates extracted clinical evidence against compiled policy rules and produces approve/escalate decisions with full audit trail.

What the PDF Contains (Policy Structure)

A typical payer policy PDF includes:

  • Policy Identifiers — Policy number (e.g., POL-RAD-402), effective dates, revision history
  • Covered Procedures — CPT/HCPCS codes with descriptions (e.g., 73721 = MRI Lower Joint)
  • Medical Necessity Criteria — Specific conditions that must be documented:
    • Minimum symptom duration (e.g., "persistent pain for at least 6 weeks")
    • Conservative therapy requirements (e.g., "6 weeks of physical therapy")
    • Required diagnostic workup (e.g., "plain radiographs completed")
    • Objective clinical findings (e.g., "joint line tenderness documented")
  • Diagnosis Restrictions — Allowed ICD-10 codes that map to this procedure
  • Step Therapy Requirements — Failed-first treatments before escalation to advanced procedures
  • Provider Requirements — Specialist consultations, network status, credentialing

Why the PDF Matters for This Demo

The PDF demonstrates that AI prior authorization governance is grounded in real clinical policy — not invented rules. Medical directors can reference the original PDF, update the markdown declarations to match policy changes, recompile, and the agent immediately enforces the updated criteria. This creates a traceable chain: PDF → Markdown → Rego → Agent Decision, where every automated decision can be audited back to the source payer policy document.

🪝 Hooks — Lifecycle Event Orchestration

Hooks are the orchestration mechanism that connects rules to the agent's execution lifecycle. They fire at specific moments during the review pipeline and execute registered callbacks. This is the event-driven architecture that makes the agent composable and extensible.

Agent Lifecycle Hooks

1. on_request_received

Fires when a PA request enters the system. Triggers PHI redaction.

2. on_guidelines_loaded

Fires after clinical guidelines are fetched. Triggers code format validation.

3. on_evidence_extracted

Fires after NLP extraction from clinical notes. Validates evidence consistency.

4. on_criteria_evaluated

Fires after OPA evaluation. Applies Clinical Conservatism routing logic.

5. on_notice_generated

Fires after letter drafting. Enforces citation and plain language rules.

Hooks decouple "what happens" (skills) from "when it's validated" (rules). This means new governance requirements can be added by registering callbacks on existing lifecycle events without modifying the core agent pipeline.

🔌 MCP Server — Model Context Protocol Architecture

The Python backend (server.py) acts as an MCP-style context server — it provides the agent with structured tools and resources through a defined protocol:

Server Endpoints (Tool Surface)

ENDPOINT MCP ANALOGY FUNCTION
POST /save-file Tool: compile Saves markdown declarations and compiles them into executable Rego/JS. Runs SkillJuror audit.
POST /generate-skill Tool: create_skill Accepts plain English, parses into skill contract, generates JS, validates syntax, registers skill.
POST /reset-workspace Tool: reset Restores all declaration and compiled files to their original templates.
GET /<file> Resource: file_read Serves static files (HTML, JS, CSS, Rego, MD) as resources the frontend agent consumes.

How This Maps to MCP Concepts

  • Tools — The POST endpoints expose capabilities (compile, generate, reset) with defined input/output schemas
  • Resources — Static file serving provides the agent with policy documents, compiled rules, and skill definitions
  • Context — The server maintains workspace state (which skills are compiled, current Rego rules) that the agent reads on each execution
  • Security Boundary — SkillJuror audits every tool invocation for unsafe patterns (eval, network calls, process access) before allowing compilation

In a production MCP deployment, this server would be registered as an MCP server in .kiro/settings/mcp.json, exposing its tools to AI agents with proper authentication and auto-approval rules.

🧠 ClinicalNLP Engine — Real AI in the Pipeline

The ClinicalNLP Engine is a local large language model (powered by Google's Gemma 4 12B) running entirely on your machine via Ollama. It replaces the regex-based clinical note extraction with genuine semantic understanding — the only component in this demo that performs real AI inference.

What It Does

When enabled (via the toggle below the Run button), the ClinicalNLP Engine reads clinical notes and extracts structured medical evidence by understanding language rather than matching patterns. It handles:

  • Negation Detection — "No physical therapy completed" → therapy = 0 weeks (regex incorrectly matches "physical therapy" as present)
  • Paraphrase Understanding — "Plain films of the joint were obtained" → hasRadiographs = true (regex only matches "radiograph" or "x-ray")
  • Contextual Reasoning — "Patient tried NSAIDs for the first two months before switching to PT last month" → therapy = 4 weeks (regex can't chain temporal references)
  • Implicit Inference — "Dr. Evans at the Rheumatology Center reviewed the case" → isRheumatologist = true (regex only matches the literal word)

Regex vs ClinicalNLP Comparison

The UI shows a side-by-side comparison when ClinicalNLP is active. Mismatches are highlighted with an accuracy insight panel that explains why the AI answer is more accurate:

CLINICAL TEXT REGEX RESULT CLINICALNLP RESULT CORRECT
"No physical therapy completed yet" 6 weeks ✗ 0 weeks ✓ ClinicalNLP
"No plain radiographs performed" Yes ✗ No ✓ ClinicalNLP
"Pain for 8 weeks with 6 weeks of PT" 8 wks / 6 wks ✓ 8 wks / 6 wks ✓ Both agree
"Consulted by Dr. Evans (Rheumatology)" Yes ✓ Yes ✓ Both agree

Architecture

  • Model: Google Gemma 4 12B (Apache 2.0, 7.6 GB quantized)
  • Runtime: Ollama (local inference on Apple Silicon unified memory)
  • Endpoint: POST /ai-extract sends clinical notes → receives structured JSON
  • Privacy: 100% local — no data leaves your machine. No cloud API calls.
  • Latency: ~3-8 seconds per extraction (vs instant for regex)
  • Progressive Disclosure: The LLM is invoked at Stage 3 only — after coverage is confirmed. It never sees the full policy corpus.

🧬 Progressive Disclosure — Agent Execution Pattern (Not Just UI)

Progressive disclosure in PriorAuthAI is NOT merely a UI presentation pattern. It is the core agent execution architecture. The agent itself receives rules, policies, and context incrementally at each pipeline stage — never the full corpus at once. This prevents context overload and mirrors how a human reviewer works: you don't read the entire policy manual before looking at a case.

Agent-Level Progressive Disclosure (Execution Architecture)

At the code level (agent.js), each pipeline stage explicitly discloses only the rules and context needed for that specific gate:

STAGE WHAT IS DISCLOSED WHAT IS WITHHELD WHY
1. Intake Raw request + PHI redaction rule only Clinical policies, Rego rules, guidelines Agent must sanitize data before it sees any policy — prevents PHI leaking into decision logs
2. Coverage Member eligibility + code format rules Clinical criteria, Rego evaluation logic No point evaluating clinical necessity if the patient isn't even covered — fail fast
3. Evidence Guidelines for THIS specific CPT code only Rego evaluation rules, decision conservatism rules Agent extracts evidence guided by the specific policy criteria — not the full guidelines catalog
4. Evaluation OPA Rego policy rules (loaded NOW, not before) Notice formatting rules, plain language rules Rego rules need extracted evidence as input — loading them earlier would be premature
5. Decision Conservatism guardrails + notice generation rules Nothing withheld — all context now available Output formatting rules are disclosed last so evaluation can't be biased by output format

Why This Matters for AI Agents

  • Prevents Context Overload — LLM-based agents degrade when given too much context at once. Disclosing incrementally keeps each decision focused and accurate.
  • Enforces Separation of Concerns — A rule scoped to Stage 5 CANNOT influence Stage 2 processing. This is architecturally enforced, not just convention.
  • Enables Early Termination — If Stage 2 fails (coverage denied), the agent never receives the Rego rules. Resources aren't wasted evaluating irrelevant policies.
  • Mirrors Human Clinical Workflow — A real reviewer checks coverage before reading guidelines. They don't pull up the notice template until they've made their decision.
  • Improves Auditability — The trace shows exactly WHEN each piece of context was disclosed. Regulators can verify the agent didn't have access to information it shouldn't have at a given stage.

UI-Level Progressive Disclosure (Presentation Layer)

The UI mirrors the agent's internal progressive disclosure with multiple depth levels for human consumers:

1
Pipeline Flowchart (Top-Level View)

The 5-stage gate flowchart (PHI Scrub → Coverage → OPA Rego → Decision → Notice) gives an instant high-level status. Green = passed, orange = warning, red = failed. Users see the overall outcome without reading any details.

2
Timeline Events (Mid-Level Detail)

The Agent Lifecycle Timeline shows each step as a compact card with type badge (hook/skill/rule/system), timestamp, and one-line summary. Users can scan the sequence without reading details.

3
Expandable Details (Deep-Level Data)

Clicking any timeline item expands it to show the full JSON payload — extracted evidence values, Rego evaluation inputs, criteria match results. This raw data is hidden until explicitly requested.

4
Decision Notice (Actionable Output)

The generated letter is the final artifact — a formal clinical decision document with policy citations and criteria checklist. It's shown after all processing completes.

Why Progressive Disclosure Is Critical for Healthcare AI Governance

  • Regulatory Compliance — Auditors need to verify specific decision steps. Progressive disclosure lets them drill into exactly the step they're investigating without sifting through irrelevant data.
  • Cognitive Load Management — Clinical reviewers handle hundreds of cases daily. A glanceable flowchart tells them instantly whether a case needs attention, without reading 20+ trace events.
  • Trust Building — When AI makes clinical decisions, transparency isn't optional. Showing "how" the decision was made at every granularity level builds trust with providers and patients.
  • Debugging & Iteration — When a policy produces unexpected results, developers can expand the exact Rego evaluation step to see which criterion failed and what the actual input values were.
  • Multi-Audience Design — Medical Directors see the flowchart. Clinical staff read the timeline. Engineers inspect the JSON payloads. Same data, different depth levels.

🤖 AVI — Your Clinical Intelligence Assistant

AVI is the conversational AI assistant embedded in PriorAuthAI. It appears as a floating purple button (🤖 AVI) in the bottom-right corner. AVI understands the current case context and can answer questions about decisions, policies, rules, and documentation requirements in real-time.

What AVI Can Do

ASK AVI WHAT IT DOES
"Why was this case escalated?" Explains the specific criteria that weren't met and which rule triggered escalation
"What's missing for approval?" Lists the exact documentation the provider needs to submit
"Explain RULE-02" Describes what Clinical Conservatism means and how it affects decisions
"What does CPT 73721 mean?" Interprets procedure and diagnosis codes in plain language
"How does progressive disclosure work?" Explains the 5-stage pipeline and how context is loaded incrementally
"What should the provider include?" Suggests specific clinical documentation for successful authorization

AVI is context-aware — it automatically sees the current case details, decision status, and clinical notes when answering. You don't need to re-explain the case.

⚡ Generative AI Features (ClinicalNLP Engine Menu)

The action buttons on the Decision Notice card and form area provide one-click access to AI-powered features. All are powered by the local ClinicalNLP Engine and process data entirely on your machine.

🧠 Rewrite with AI

Takes the auto-generated decision notice letter and rewrites it in warm, empathetic, patient-friendly language. Preserves all factual content (policy references, criteria, decision) but transforms clinical jargon into plain English. Useful for patient-facing correspondence where tone matters.

When to use: After running a review, click to transform the template letter before sending to the patient.

📋 Generate Appeal Guide

When a case is escalated, this generates a specific, actionable appeal guide for the provider. It identifies exactly what documentation is missing, which criteria weren't met, what the provider should submit, and suggests language for the appeal letter. Prepended above the original notice.

When to use: After an escalation decision. Gives providers a clear path to approval.

📝 Summarize Notes

Condenses the clinical notes field into exactly 3 bullet points capturing the key medical facts relevant to prior authorization. Designed for quick reviewer triage — a clinical director can glance at the summary instead of reading the full notes. Appears as a popup card above the notes textarea.

When to use: Before or after running a review. Helpful for cases with lengthy clinical documentation.

📊 Score Notes Quality

Scores the clinical notes on a 1-10 scale for completeness against prior authorization requirements. Identifies specific missing elements (e.g., "symptom duration not documented", "no imaging results mentioned") and suggests improvements. Displayed as a color-coded popup: green (7-10), orange (4-6), red (1-3).

When to use: Before submitting a case. Helps providers self-assess if their documentation is sufficient.

🧠 ClinicalNLP Engine Toggle

The checkbox below the Run button switches the clinical note extraction from regex-based pattern matching to full semantic AI extraction. When active, the ClinicalNLP Engine reads the notes with language understanding — detecting negation ("No PT completed" → 0 weeks), paraphrases, and implicit references that regex cannot handle. Shows a side-by-side comparison with accuracy insights.

When to use: Always recommended for complex or ambiguous clinical notes. Toggle off for instant (but less accurate) processing.

How All AI Features Connect to the Architecture

Every AI feature follows the same governance pattern as the core agent:

  • Progressive Disclosure — AI features only receive the context they need. AVI gets the current case but not raw PHI. The summarizer gets notes but not member IDs.
  • Audit Trail — Every AI invocation is logged in the timeline trace with the engine name and response type.
  • Local Execution — All AI processing happens on your machine via Ollama. No data leaves the device. HIPAA-compatible by design.
  • Graceful Degradation — If the ClinicalNLP Engine is offline (Ollama not running), all features show clear error messages and the system falls back to regex-based processing.
  • MCP Pattern — The POST /ai-chat endpoint acts as an MCP tool with defined input (prompt + context) and output (response text), following the same contract pattern as other server tools.

🔁 Challenger Agent — Autonomous Quality Review

The Challenger Agent is a separate autonomous agent that independently reviews every decision made by the PA Review Agent — both approvals AND escalations. It operates with its own skills, rules, and hooks.

Does the Challenger Ever Agree?

Yes. For cases where the evidence clearly supports the decision (e.g., the Dupixent case with EASI score 28 and 3 documented failed medications), the Challenger agrees with high confidence. It only challenges on borderline or weak-evidence cases.

  • ✅ AGREE — Documentation is thorough. Quality confirmed. No override.
  • ⚠️ CONCERN — Minor gaps noted but not enough to override (confidence < 7).
  • 🚩 CHALLENGE — Formal override. Decision flagged for Medical Director review (confidence ≥ 7).

Why Two Agents? (If the First Should Get It Right)

The two agents have fundamentally different objectives by design:

🤖 PA Agent 🔁 Challenger Agent
Goal Make a decision efficiently Stress-test that decision
Bias Conservative (RULE-02: never deny) Adversarial (look for problems)
Perspective "Does evidence meet criteria?" "Is the evidence actually strong enough?"
Scope Evaluates what's IN the notes Looks for what's MISSING or contradictory
Speed Fast, threshold-based Slower, reasoning-based

The PA Agent isn't "wrong" — it correctly applies rules. But it uses pattern matching and threshold checks. The Challenger uses reasoning to ask: "yes the keyword matched, but is the underlying evidence actually solid?"

Real-World Parallels

This is the same proven pattern used in:

  • Audit + Primary Accounting — one does the work, one checks it
  • Maker/Checker in Banking — dual authorization for transactions
  • Red Team / Blue Team — adversarial security testing
  • Peer Review in Academia — independent validation of findings

The first agent is optimized for throughput (fast decisions). The second is optimized for accuracy (catching edge cases). Together they are better than either alone.

🚀 Quick Start Guide

Running a Prior Auth Review

  1. Select a preset case (1-4) from the middle column to populate the form
  2. Review or edit the patient details, CPT/ICD codes, and clinical notes
  3. Toggle guardrail rules on/off using the checkboxes to see their effect
  4. Click ⚡ Run Prior Auth Agent Review
  5. Watch the pipeline flowchart light up as each stage completes
  6. Click timeline items to expand and see detailed JSON payloads
  7. Read the generated decision notice letter at the bottom

Editing Policies

  1. Click the rules_declaration.md tab in the left column
  2. Edit criteria values (e.g., change "Minimum Symptom Duration: 6 weeks" to "4 weeks")
  3. Click 💾 Save & Compile Policies
  4. The SkillJuror audit console shows compilation results
  5. Switch to the rules.rego tab to see the compiled output
  6. Re-run a case to see the updated criteria in action

Adding a New Skill

  1. Find the "Create Skill from Plain English" panel at the bottom of the left column
  2. Type a description like: "Check if patient age is over 65 and flag for senior review"
  3. Click 🧠 Generate & Register Skill
  4. The skill appears as a new pill in the "Discovered Active Skills" list
  5. Run a case — the new skill executes and appears in the timeline trace
